<p>Cognitive walkthroughs are used to examine the <a href="https://www.interaction-design.org/literature/topics/usability" title="Usability">usability</a> of a product. They are designed to see whether or not a new user can easily carry out tasks within a given system. It is a task-specific approach to usability (in <a href="https://www.interaction-design.org/literature/topics/contrast" title="Contrast">contrast</a> to <a href="https://www.interaction-design.org/literature/topics/heuristic-evaluation" title="Heuristic Evaluation">heuristic evaluation</a> which is a more holistic usability inspection). The idea is that if given a choice – most users prefer to do things to learn a product rather than to read a manual or follow a set of instructions.</p><p><img src="https://public-media.interaction-design.org/images/uploads/6b6745b67bb3b5563f451de76f83c878.png"></p><p><small><i>Author/Copyright holder: Paul Veugen. Copyright terms and licence: CC BY 2.0</i></small>
</p><p>The biggest benefit of a <a href="https://www.interaction-design.org/literature/topics/cognitive-walkthrough" title="Cognitive Walkthrough">cognitive walkthrough</a> (or walkthroughs) is that it is extremely cost-effective and fast to carry out when compared to many other forms of <a href="https://www.interaction-design.org/literature/topics/usability-testing" title="Usability Testing">usability testing</a>. It can also be implemented prior to development during the <a href="https://www.interaction-design.org/literature/topics/design" title="Design">design</a> phase which can give rapid insight before budget is spent developing an unusable product.</p><h2>How to Conduct a Cognitive Walkthrough</h2><p>A cognitive walkthrough begins by defining the task or tasks that the user would be expected to carry out. It is these tasks that the cognitive walkthrough will examine for usability – any tasks that can be performed in the product but are not subject to a cognitive walkthrough will not, normally, be assessed during the process.</p><h2>Tasks Become Processes</h2><p>Tasks are then divided up into a simple process to follow. So, for example, the log in process on a website might look like this:</p><ul> <li>Open browser</li> <li>Navigate to site</li> <li>Click login button</li> <li>Enter user name in user name field</li> <li>Enter password in password field</li> <li>Click the login button</li>
</ul><p>If the task is too complex to write in a <a href="https://www.interaction-design.org/literature/topics/list" title="List">list</a> format – a diagram can be used instead.</p><p><img src="https://public-media.interaction-design.org/images/uploads/bbf7bbc8b5dc7a5650ba828f973f767a.png"><small><i>Author/Copyright holder: Raylton P. Sousa. Copyright terms and licence: CC BY -SA 3.0</i></small> <br>
</p><h2>The Four Questions to be Asked during a Cognitive Walkthrough</h2><p>Blackmon, Polson, et al. in 2002 in their paper “Cognitive walkthrough for the Web” offer four questions to be used by an assessor during a cognitive walkthrough:</p><ul> <li>Will the user try and achieve the right outcome?</li> <li>Will the user notice that the correct action is available to them?</li> <li>Will the user associate the correct action with the outcome they expect to achieve?</li> <li>If the correct action is performed; will the user see that progress is being made towards their intended outcome?</li>
</ul><p>The assessor performs each action in any given task process and asks the four questions above. Let’s take a look at each question more closely and see what they’re trying to achieve:</p><h3>Will the user try and achieve the right outcome?</h3><p>This question is trying to examine whether the interface is making <a href="https://www.interaction-design.org/literature/topics/assumptions" title="Assumptions">assumptions</a> about a user’s level of experience or knowledge that aren’t accurate. It can also help identify when a user’s <a href="https://www.interaction-design.org/literature/topics/expectations" title="Expectations">expectations</a> of an action don’t align with the actual action taken because they are using other reference points and becoming confused (for example, the use of language in your product is in common usage in other products and means something else).</p><p>For example; let’s say you have invented a new gun. Your gun is fired by pushing a button on the side and the traditional trigger mechanism is used to chamber a round before firing. It is unlikely that your gun will be a success because in all other guns – a trigger is used to fire a round. The user is likely to pull the trigger and be disappointed by the outcome.</p><p><img src="https://public-media.interaction-design.org/images/uploads/ee3b4b803bc6f0ecd5fc983474f979e0.jpg" width="567" height="568" style="width: 567px; height: 568px;"></p><p><small><i>Author/Copyright holder: Gibraltar Arms. Copyright terms and licence: CC BY-SA 3.0</i></small>
</p><h3>Will the user notice that the correct action is available to them?</h3><p>Hidden or obscured controls are a problem for users. The more data that you present and the more choice that you present; the less likely it is that a user will know what to do. If you bury a control in a menu system rather than having it present while an action is taking place, the same is true; your users won’t know what to do.</p><p>One of the most common examples of this in real life is the TV or DVD remote control. These provide an overwhelming number of options to users and are often confusing to use. In many cases manufacturers try to tackle this by placing rarely used functions underneath a sliding cover; by reducing options the control is easier to use.</p><h3>Will the user associate the correct action with the outcome they expect to achieve?</h3><p>If your use of language is poor, for example you use overly complex words or industry jargon, it can be hard for a user to work out what is needed to achieve their outcome. This is also true when you use complex actions (think Ctrl+Alt+Del – it may be something you’re intimately familiar with now – if you use Windows – but it’s not exactly intuitive is it?) to carry out an action.</p><p>Steve Krug, says in his book “Don’t Make me Think” – “It doesn’t matter how many times I have to click, as long as each click is a mindless, unambiguous choice.”</p><h3>If the correct action is performed; will the user see that progress is being made towards their intended outcome?</h3><p>This will help you investigate when <a href="https://www.interaction-design.org/literature/topics/feedback" title="Feedback">feedback</a> in the system is missing, badly worded, easy to miss or plain old ambiguous. You need to let your users know about the progress they are making in the task. It’s why computer <a href="https://www.interaction-design.org/literature/topics/games" title="Games">games</a> use loading screens to signify moving on to the next level and why it’s always a good idea to have a “thank you” note when a customer finishes making an order.</p><h2>Who Should Conduct a Cognitive Walkthrough?</h2><p>Anyone can conduct a cognitive walkthrough; however, there is a risk that someone who is already familiar with your jargon, language and system is going to miss things that someone who lacks that familiarity would find.</p><p>Given that conducting a cognitive walkthrough is not particularly challenging. If you’ve prepared the task lists and can copy the questions (plus a little explanation as to how to use them) on to a piece of paper – you should be able to find someone in your organization who can do it. You can also ask users to conduct these walkthroughs but that will add <a href="https://www.interaction-design.org/literature/topics/recruitment" title="Recruitment">recruitment</a> time and <a href="https://www.interaction-design.org/literature/topics/expense" title="Expense">expense</a> (you’ll need to compensate the user for their time).</p><p>If you have to use someone who is very familiar with the product, make sure they have <a href="https://www.interaction-design.org/literature/topics/user-personas" title="User Personas">user personas</a> to <a href="https://www.interaction-design.org/literature/topics/hand" title="Hand">hand</a> – to try and guide them to “walk a mile in the user’s shoes”.</p><p><img src="https://public-media.interaction-design.org/images/uploads/6e6d459d0c46a391f1671e30ace3f1df.jpg" width="707" height="530" style="width: 707px; height: 530px;"></p><p><small><i>Author/Copyright holder: Walton LaVonda. Copyright terms and licence: Public Domain.</i></small></p><h2>What Do You Do with the Answers to the Questions?</h2><p>Each assessor involved in a cognitive walkthrough should record the step in the process where they found an issue and what that issue was. This can be done in their own words.</p><p>When the process is complete, it’s a good idea to roundup all the assessors’ reports into a single report and then prioritize <a href="https://www.interaction-design.org/literature/topics/issues" title="Issues">issues</a> for fixing.</p><h2>Cognitive Walkthroughs and Information Visualization</h2><p>An emerging design field in which cognitive walkthroughs are being used is <a href="https://www.interaction-design.org/literature/topics/information-visualization" title="Information Visualization">information visualization</a>. In this discipline the questions you ask might need to be tweaked based on the type of visualization being employed and the complexity of the data. Unfortunately, as yet there are no “ideal” questions formulated for cognitive walkthroughs of information visualizations so if you intend to use the technique – you’ll need to <a href="https://www.interaction-design.org/literature/topics/experiment" title="Experiment">experiment</a> to find the right questions to bring results for your users.</p><h2>The Take Away:</h2><p>Cognitive walkthroughs are cheap and easy to conduct. They allow the system or product to be examined for usability from a user’s perspective and provide quick feedback to enable <a href="https://www.interaction-design.org/literature/topics/decision-making" title="Decision Making">decision making</a> in the <a href="https://www.interaction-design.org/literature/topics/design-process" title="Design Process">design process</a>. A product does not need to have been developed to conduct a cognitive walkthrough – the walkthrough can take place using <a href="https://www.interaction-design.org/literature/topics/sketches" title="Sketches">sketches</a> or other forms of <a href="https://www.interaction-design.org/literature/topics/prototype" title="Prototype">prototype</a>. </p><h2>References:</h2><p>Blackmon, M. H. Polson, P.G. Muneo, K & Lewis, C. (2002) Cognitive Walkthrough for the Web CHI 2002 vol.4 No.1 pp463–470</p><p>Cognitive walkthrough approach proposed at Chi 95 - <a href="http://www.sigchi.org/chi95/proceedings/tutors/jr_bdy.htm">http://www.sigchi.org/chi95/proceedings/tutors/jr_bdy.htm</a>
</p><p><em>Hero Image: Author/Copyright holder: Nicola Jones. Copyright terms and licence: CC BY-ND 2.0</em><em></em><br></p>
